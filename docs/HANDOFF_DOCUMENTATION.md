# Мини-пакет документации для хенд-оффа: Dish Analyzer (Анализатор блюд)

---

## 1. Project Overview

### Цель проекта
Автоматизировать получение структурированной информации о блюде по фотографии: распознавание блюда и ингредиентов, генерация пошагового рецепта с КБЖУ/ХЕ и опциональная визуализация «идеального» вида блюда.

### Целевой пользователь
- Домашние пользователи и любители готовить (быстрый рецепт по фото).
- Пользователи, следящие за питанием (КБЖУ, хлебные единицы).
- Разработчики/студенты как пример пайплайна Vision → LLM → Image Gen.

### Ключевая ценность
- **Один вход** — фото блюда; **один выход** — анализ + рецепт + КБЖУ + опциональное изображение.
- Использование российского GigaChat для рецепта и OpenAI для анализа изображения и генерации картинки.
- Два интерфейса: CLI для скриптов/автоматизации и веб-интерфейс (Flask) для интерактивного использования.

### Стек и поток данных
| Этап | Сервис | Назначение |
|------|--------|------------|
| 1 | OpenAI Vision (gpt-4o) | Анализ изображения → JSON (название, ингредиенты, кухня, confidence) |
| 2 | GigaChat-2 | Генерация рецепта, этапов, КБЖУ/ХЕ и промпта для изображения (JSON) |
| 3 | OpenAI DALL-E | Генерация изображения по промпту от GigaChat |

## 2. Prompt Engineering Guide (черновик)

### Промпт 1: Анализ изображения блюда (OpenAI Vision)

**Назначение:** по одному изображению вернуть структурированный JSON с названием блюда, списком ингредиентов, типом кухни и уверенностью (0–1).

**Расположение в коде:** `dish_analyzer.py`, функция `analyze_dish()`, переменная `prompt` (строки ~80–95).

**Текущая логика промпта:**
- Явно задан целевой JSON-шаблон с полями: `dish_name`, `ingredients`, `cuisine`, `confidence`.
- Уточнения: язык (русский), диапазон confidence 0.0–1.0, требование вернуть только валидный JSON без комментариев.

**Параметры API (настраиваются через .env):**
- `OPENAI_MODEL` — модель (по умолчанию `gpt-4o`).
- `OPENAI_MAX_TOKENS` — лимит токенов (по умолчанию 500).
- `OPENAI_TEMPERATURE` — температура (по умолчанию 0.3; низкая для стабильного JSON).

**Ограничения и когда работает плохо:**
- Размытое/тёмное фото, несколько блюд в кадре, нестандартная подача — возможны ошибки в названии и списке ингредиентов.
- Модель иногда оборачивает JSON в markdown (`` ```json ``); в коде это обрабатывается (извлечение блока между `` ``` ``).
- Сильно нестандартные или региональные блюда могут дать неточную кухню или название.
- Максимальный размер/разрешение изображения ограничены возможностями Vision API.

**Рекомендации по доработке промпта:**
- Добавить пример 1–2 строк «хорошего» ответа для few-shot при частых ошибках формата.
- Явно указать: «Не включай в ответ ничего кроме JSON» для снижения мусора перед/после JSON.

### Промпт 2: Генерация рецепта и КБЖУ (GigaChat-2)

**Назначение:** по данным анализа блюда (название, ингредиенты, кухня) сгенерировать полный рецепт в JSON: метаданные рецепта, список ингредиентов с количествами, пошаговые этапы, КБЖУ и ХЕ (на порцию и на 100 г), а также текстовый промпт на английском для DALL-E.

**Расположение в коде:** `dish_analyzer.py`, функция `generate_recipe_with_gigachat()`, переменная `prompt` (строки ~409–441).

**Текущая логика промпта:**
- В промпт подставляются: `dish_name`, `ingredients`, `cuisine` из результата Vision.
- В теле промпта задана полная JSON-схема с вложенными объектами: `recipe`, `cooking_steps`, `nutrition` (per_serving, per_100g), `image_prompt`.
- Оговорено: текст на русском, кроме `image_prompt` (английский); расчёт ХЕ = углеводы / 12; только валидный JSON, без markdown-блоков.

**Параметры API (через .env):**
- `GIGACHAT_MODEL`, `GIGACHAT_MAX_TOKENS`, `GIGACHAT_TEMPERATURE`, `GIGACHAT_REPETITION_PENALTY`.
- По умолчанию: max_tokens=4000, temperature=0.7, repetition_penalty=1.1.

**Ограничения и когда работает плохо:**
- Ошибки анализа (Vision) передаются в GigaChat — «мусор на входе» ведёт к неточностям в рецепте и КБЖУ.
- GigaChat иногда возвращает JSON в markdown; в коде выполняется срез по `` ```json `` / `` ``` ``.
- КБЖУ и ХЕ — оценочные; не заменяют профессиональный диетологический расчёт.
- При слишком общем описании блюда рецепт может быть шаблонным; при редких блюдах — галлюцинации в этапах или ингредиентах.
- Длинные ответы при большом числе шагов могут обрезаться по `max_tokens`.

**Рекомендации по доработке промпта:**
- Зафиксировать число порций по умолчанию (например, 4) и явно указать в промпте.
- Добавить в инструкцию явное требование: «Сначала рассчитай КБЖУ, затем bread_units = carbs/12».
- При частых разрывах JSON — увеличить `GIGACHAT_MAX_TOKENS` или разбить ответ на два запроса (рецепт и отдельно КБЖУ).

## 3. Метрики успеха

### Технические метрики
- **Доля успешных прогонов пайплайна:** % запросов, в которых без ошибки получены и анализ, и рецепт (и при включённой опции — изображение).
- **Валидность JSON:** % ответов Vision и GigaChat, которые парсятся без fallback по извлечению из markdown.
- **Время отклика:** среднее время по этапам (Vision, GigaChat, DALL-E) и суммарное для CLI/веб (целевой ориентир — приемлемо для интерактивного веб, например &lt; 60–90 с на полный цикл).
- **Доступность внешних API:** доля успешных вызовов OpenAI и GigaChat (в т.ч. по кодам ответа и таймаутам).

### Бизнес/пользовательские метрики
- **Релевантность анализа:** совпадение названия/кухни с ожиданиями пользователя (выборочная разметка или опрос).
- **Полезность рецепта:** готовность пользователя приготовить по рецепту (опрос, лайки/дизлайки в UI).
- **Корректность КБЖУ/ХЕ:** сравнение с эталонными значениями для известных блюд (выборочная проверка).
- **Удовлетворённость сгенерированным изображением:** субъективная оценка «похоже на блюдо» (опционально).

## 4. Основные риски и ограничения

1. **Зависимость от внешних API и ключей**  
   Работа приложения возможна только при наличии и корректности `OPENAI_API_KEY` и (для рецепта) `GIGACHAT_API_KEY`. Изменение тарифов, лимитов или отключение API сделает продукт частично или полностью нерабочим. Рекомендуется документировать минимально необходимые квоты и иметь fallback-сценарий (например, только анализ без рецепта при отсутствии GigaChat).

2. **Качество входного изображения и границы Vision**  
   Результат сильно зависит от качества фото (освещение, ракурс, один объект в фокусе). Нестандартные блюда, смешанные тарелки или плохое качество ведут к ошибкам в названии и ингредиентах, которые далее тиражируются в рецепте и КБЖУ. Ограничение по размеру/формату файла (в т.ч. в веб-интерфейсе) также накладывает предел на разрешение.

3. **Оценочный характер рецепта и КБЖУ**  
   Рецепт и расчёты КБЖУ/ХЕ носят справочный характер и не являются медицинскими или диетологическими рекомендациями. Риск — использование данных при диабете или строгих диетах без консультации специалиста. В продукте целесообразно явно указывать, что КБЖУ и ХЕ приблизительные (например, в UI и в документации).
